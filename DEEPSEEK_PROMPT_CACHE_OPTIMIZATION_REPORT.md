# 群聊刺客 DeepSeek 前缀缓存优化与真实群聊测试报告

## 1. 文档信息

- 插件：`OlivOSAIChatAssassin`
- 测试日期：2026-07-11
- 测试版本：`1.4.3`
- SVN：`63`
- 接口类型：OpenAI 兼容 `/chat/completions`
- 目标模型：DeepSeek 系列模型
- 测试数据：项目中的真实群聊样本 `test_real_chat_12_context.py`
- 对话消息数：72 条
- 历史最低窗口：12 条
- 优化后增长上限：32 条

## 2. 结论摘要

这次缓存命中率偏低的核心原因不是 DeepSeek 缓存失效，而是旧版插件每收到一条新消息就滚动一次固定历史窗口。

当历史达到 12 条后，旧逻辑会在每次追加消息时删除最旧的一条。请求中的第一条群聊历史因此持续变化。DeepSeek 的缓存依赖请求开头存在完全相同的 token 前缀，一旦靠前的历史消息变化，后面即使仍有 11 条相同消息，也不能继续作为连续前缀命中。

优化后，历史从 12 条开始连续增长到 32 条，达到上限后才批量保留最近 11 条并加入新消息。这样大多数请求只会在末尾增加内容，已有系统提示词和历史前缀不会移动。

使用本项目保存的 72 条真实群聊进行结构测试，结果如下：

| 方案 | 历史策略 | 计入请求 | 历史大幅换代 | 请求前缀结构复用率 |
|---|---:|---:|---:|---:|
| 优化前 | 固定 12 条 | 67 | 60 次 | 40.72% |
| 优化后 | 12 条增长至 32 条 | 67 | 2 次 | 92.55% |
| 激进配置 | 12 条增长至 64 条 | 67 | 1 次 | 95.25% |

默认采用 32 条上限。它比 64 条输入更短，仍然能消除绝大多数逐条滚动造成的缓存断裂。

> 注意：表中的复用率是对连续请求 JSON 进行 UTF-8 字节级最长相同前缀测量，不是 DeepSeek 服务端返回的最终 token 缓存账单。实际 `prompt_cache_hit_tokens` 还会受到模型分词、服务端缓存有效期、缓存淘汰、不同 API Key、不同模型、动态记忆和技能片段变化影响。

## 3. 原始问题分析

截图中的统计为：

| 项目 | Token 数 |
|---|---:|
| 输入命中缓存 | 27,136 |
| 输入未命中缓存 | 75,287 |
| 输出 | 2,432 |

只计算输入部分：

```text
输入总量 = 27,136 + 75,287 = 102,423
缓存命中率 = 27,136 / 102,423 ≈ 26.49%
```

旧请求结构大致如下：

```text
system: 固定规则、人格、QQ号
user: 历史消息 1
user: 历史消息 2
...
user: 历史消息 12
user: 当前时间、当前记忆、技能片段、图片缓存
user: 最终输出要求
```

加入一条新消息后，旧固定窗口变成：

```text
system: 固定规则、人格、QQ号
user: 原历史消息 2
user: 原历史消息 3
...
user: 原历史消息 12
user: 新消息 13
user: 新的动态上下文
user: 最终输出要求
```

虽然中间仍有大量相同历史，但第一条历史已经由“消息 1”变为“消息 2”，请求的连续相同前缀在这里中断。因此通常只有前面的系统提示词能够命中。

## 4. 实施的优化

### 4.1 新增缓存优化配置

默认配置增加：

```json
{
  "prompt_cache_optimized": true,
  "prompt_cache_history_size": 32
}
```

配置含义：

- `prompt_cache_optimized`：是否启用前缀缓存友好的历史窗口。
- `prompt_cache_history_size`：历史连续增长的最大条数。
- `history_size`：批量换代后保留的最低历史数量。

默认行为是：

```text
8 → 9 → 10 → ... → 31 → 32 → 8 → 9 → ...
```

本报告为了复现之前的真实群聊测试，使用的是：

```text
12 → 13 → 14 → ... → 31 → 32 → 12 → 13 → ...
```

### 4.2 统一历史窗口计算

历史文件恢复、新群初始化和运行日志现在统一调用 `load.get_history_limits()`。

该函数会：

1. 校验 `history_size`，防止无效字符串或小于 1 的数值。
2. 在缓存优化开启时使用 `prompt_cache_history_size`。
3. 确保增长上限不小于最低历史数量。
4. 关闭缓存优化后兼容原有 `history_dynamic` 行为。

### 4.3 优化记忆总结请求

旧版记忆总结会把全部历史合并为一个巨大 `user` 字符串：

```text
现在提炼如下对话中的重要知识点：
消息 1
消息 2
...
前情提要：动态内容
```

只要新增或删除一条消息，这整个字符串都会改变，通常只能复用系统提示词。

现在改成逐条消息结构：

```text
system: 记忆总结规则
user: 历史消息 1
user: 历史消息 2
...
user: 当前前情提要
user: 最终总结要求
```

历史增长期间，旧消息保持在相同位置，动态前情提要和最终任务位于末尾，不会破坏前面的可缓存历史。

### 4.4 保留动态内容在历史之后

当前本地时间、相关记忆、技能片段和图片缓存仍放在历史消息之后。这一点是正确的，因为这些内容可能每次请求都变化。

请求顺序保持为：

```text
稳定系统提示词
稳定或连续增长的群聊历史
动态上下文
本次输出要求
```

动态内容即使未命中，也不会反向破坏系统提示词和历史前缀的缓存。

## 5. 真实群聊测试方法

### 5.1 测试数据

测试直接使用 `test_real_chat_12_context.py` 中的 72 条真实群聊消息，没有随机生成对话文本。

对话包含以下连续话题：

- 灵魂武器碎裂后再次召唤；
- 是否消耗灵力；
- 同一场景是否增加消耗；
- 灵能力激活多个标签和选择多个目标；
- 属性检定、防御、反击和豁免；
- 战斗轮双方对抗检定的成功等级；
- 群友对机器人回答错误的连续追问。

### 5.2 对比配置

优化前：

```json
{
  "history_size": 12,
  "history_dynamic": false,
  "prompt_cache_optimized": false
}
```

优化后：

```json
{
  "history_size": 12,
  "prompt_cache_optimized": true,
  "prompt_cache_history_size": 32
}
```

额外观察配置：

```json
{
  "history_size": 12,
  "prompt_cache_optimized": true,
  "prompt_cache_history_size": 64
}
```

### 5.3 请求构造

每条群聊消息使用与插件一致的历史字段：

```json
{
  "timestamp": 1783770000,
  "time": "2026-07-11T20:28:59+08:00",
  "user_id": "100000",
  "nickname": "小小怪将军",
  "message": "灵魂武器碎了再召唤要多少蓝来着"
}
```

每轮请求均包含：

1. 稳定系统提示词；
2. 当前历史队列；
3. 每轮变化的本地时间；
4. 每轮变化的当前记忆；
5. 每轮变化的技能片段；
6. 固定最终输出要求。

因此测试不会通过把动态内容伪装成固定内容来抬高结果。

### 5.4 测量方法

对相邻两次请求执行以下步骤：

1. 使用确定性 JSON 序列化请求消息。
2. 编码为 UTF-8 字节。
3. 从请求开头逐字节比较。
4. 遇到第一个不同字节时停止。
5. 将此前完全相同的字节记为“可复用请求前缀”。

加权复用率计算方式：

```text
加权复用率 = 全部相邻请求可复用前缀字节 / 全部被测请求输入字节
```

该方法能够准确反映“请求前部是否因历史滚动而提前变化”，但不代替 DeepSeek 服务端 token 统计。

## 6. 测试结果

### 6.1 优化前：固定 12 条

```text
群聊消息：72
计入测量请求：67
累计请求输入：260,641 bytes
累计可复用前缀：106,143 bytes
加权前缀复用率：40.7238%
相邻请求平均复用率：40.8126%
历史换代次数：60
最终历史长度：12
```

从第 13 条消息开始，每条新消息都会触发一次滚动。也就是说，在历史达到上限后，绝大多数请求只能复用系统提示词，无法继续复用历史消息前缀。

### 6.2 优化后：12 条增长至 32 条

```text
群聊消息：72
计入测量请求：67
累计请求输入：378,715 bytes
累计可复用前缀：350,484 bytes
加权前缀复用率：92.5456%
相邻请求平均复用率：91.6869%
历史换代次数：2
最终历史长度：30
```

两次换代点为：

| 消息序号 | 时间 | 昵称 | 消息 |
|---:|---|---|---|
| 33 | 20:35:42 | 186 | 源头找到了 |
| 54 | 20:37:36 | 禺日居士 | @幻紫仙门 兑 |

除这两个批量换代点外，其他请求都可以保持此前历史作为连续前缀。

### 6.3 激进配置：12 条增长至 64 条

```text
群聊消息：72
计入测量请求：67
累计请求输入：545,631 bytes
累计可复用前缀：519,729 bytes
加权前缀复用率：95.2528%
相邻请求平均复用率：93.8631%
历史换代次数：1
最终历史长度：19
```

唯一换代点为第 65 条消息：

```text
20:39:03 陆溟：小芙真好 还送你30点灵力
```

64 条窗口比 32 条窗口的结构复用率再提升约 2.71 个百分点，但累计输入字节从 378,715 增加到 545,631，增长约 44.06%。因此不建议默认使用 64，除非缓存命中率优先级明显高于单次上下文长度。

## 9. 配置建议

### 9.1 推荐配置

适用于多数群聊：

```json
{
  "history_size": 8,
  "prompt_cache_optimized": true,
  "prompt_cache_history_size": 32
}
```

特点：

- 绝大多数连续请求可以复用已有历史；
- 批量换代频率较低；
- 上下文长度和缓存收益较平衡。

### 9.2 高缓存命中配置

适用于高频群聊、上下文限制宽松、特别重视缓存命中的情况：

```json
{
  "history_size": 12,
  "prompt_cache_optimized": true,
  "prompt_cache_history_size": 64
}
```

代价：

- 单次请求更长；
- 动态记忆、技能片段和图片缓存仍然可能未命中；
- 如果 API 缓存失效，较长历史会全部按未缓存输入计费。

### 9.3 关闭优化

如果必须严格固定历史条数：

```json
{
  "history_size": 12,
  "history_dynamic": false,
  "prompt_cache_optimized": false
}
```

这会恢复每条消息滚动一次的旧行为，DeepSeek 前缀缓存命中率通常会明显下降。

## 10. 部署与观察

### 10.1 部署步骤

1. 更新整个 `OlivOSAIChatAssassin` 目录。
2. 确认 `app.json` 版本为 `1.4.3`、SVN 为 `63`。
3. 重载插件，使内存中的历史队列按新配置重建。
4. 检查实际生效的 `config_<botHash>.json`。
5. 确认配置中存在 `prompt_cache_optimized` 和 `prompt_cache_history_size`。

旧配置文件缺少新字段时，插件的默认值合并逻辑会自动补充字段。多机器人部署时，应检查机器人专属配置文件，而不只检查公共 `config.json`。

### 10.2 日志观察

插件已经记录 DeepSeek 返回的缓存统计：

```text
USAGE - CACHE - 85.00 % (命中Token/未命中Token)
```

历史窗口日志类似：

```text
HISTORY - SIZE [18/32]
```

建议连续观察至少一个完整的 32 条窗口周期。换代请求命中率暂时下降属于正常现象，随后请求应重新逐步提高。

### 10.3 建议记录的数据

建议记录 1 至 3 天：

- `prompt_cache_hit_tokens`；
- `prompt_cache_miss_tokens`；
- 命中率；
- 每次请求历史长度；
- 是否刚发生批量换代；
- 主模型与意图模型是否使用相同模型和接口；
- 平均输入 token；
- 实际输入费用。

缓存优化的目标不只是提高百分比，还应同时比较最终费用和回复质量。

## 11. 限制与后续方向

本次优化解决的是最主要的历史滚动问题，但以下动态内容仍然天然难以命中：

- 当前本地时间；
- 当前群记忆；
- 按话题检索的知识；
- 动态技能片段；
- 当前图片缓存；
- 意图模型临时选择的图片文件名；
- 服务端缓存过期后的首次请求。

这些内容已经放在历史之后，因此不会破坏前面的稳定缓存。后续如果还要继续提高命中率，可以考虑：

1. 让本地时间只保留到分钟或按固定时间片更新；
2. 对未变化的动态记忆生成版本号，只在内容真正变化时重建；
3. 限制技能片段无关抖动，确保同一话题返回顺序稳定；
4. 在日志中增加“是否发生历史批量换代”标记；
5. 分别统计主回复、首次意图判断、记忆总结和 OCR 请求的缓存命中率。

## 13. 最终结论

这次优化解决了造成低命中率的主要结构问题：固定滑动窗口让请求前部在每条消息后发生变化。

真实群聊结构测试显示：

- 历史换代从 60 次下降到 2 次；
- 可复用请求前缀从约 40.72% 提高到约 92.55%；
- 10 个真实规则问题仍全部正确路由；
- 技能检索仍保持最近 12 条上下文，不受主模型增长窗口干扰；
- 动态时间、记忆、技能和图片内容仍被放在请求末尾。

推荐先使用默认 `prompt_cache_history_size=32` 上线观察 DeepSeek 的实际 `prompt_cache_hit_tokens`。如果真实命中率仍不足且输入成本可接受，再将上限提高到 48 或 64。
